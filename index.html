📂 一、 專案目錄結構調查報告：src 與 test 的功能與關聯

在開始撰寫程式前，建立良好的目錄結構是軟體工程的基石。在專案開發中，src 與 test 是最核心的兩個資料夾：

1. src 資料夾 (Source Code / 原始碼)

  • 主要功能： 存放維持應用程式核心運作的正式程式碼。
  • 內容範例： 網頁前端畫面（HTML/CSS/JS）、後端的業務邏輯與資料庫處理程式碼（run.py、app.py）。
  • 重要性： 這是軟體成品的本體。專案在進行編譯、打包與正式上線部署時，系統主要只會讀取此資料夾。

2. test 資料夾 (Test Code / 測試碼)

  • 主要功能： 存放專門用來檢驗 src 內程式碼是否正確執行的自動化「測試腳本」。
  • 內容範例： 檢驗單一函數的單元測試（Unit Tests）、測試多模組組合的整合測試（Integration Tests）與模擬數據。
  • 重要性： 專案的品質把關者。測試碼不會被打包給終端使用者看見，但開發人員在修改原始碼後，會執行測試來確保新功能沒有改壞舊功能。

🔄 兩者的關聯性：為了方便後續維護，test 資料夾的文件結構通常會完全模仿 src。例如：
若原始碼路徑為：src/components/calculator.js
對應測試碼就會命名為：test/components/calculator.test.js

🔒 二、 雲端環境安全強化指南：防止帳號被盜與控制天價帳單

雲端根帳號擁有最高控制權，一旦流失將造成災難性損失。以下為安全強化指南：

1. AWS 雲端根帳號 (Root User) 強化

  • 啟用多重要素驗證 (MFA)： 這是防止密碼外洩最重要防線。至 IAM 主控台登記綁定手機 Google Authenticator 等虛擬通行密鑰。
  • 刪除 Root 存取金鑰： 根帳號絕對不要建立 access_key_id。若程式碼不小心流出金鑰，駭客將能搬空你的整個雲端資源。
  • 建立群組管理： 實務上建立 Admin-Group（給予系統最高權限）與 DevOps-Group 權限組。日常維運使用新增的 IAM 使用者（如：jackyadmin）登入，並封印 Root 帳號。

2. 💰 實務控管：AWS 預算設定 4 步驟

為了徹底避免產生龐大費用，必須自行設定預算告警通知：

  1. 選擇預算類型： 選擇「使用推薦範本」，勾選「零預算金額」或「每月成本預算」。
  2. 設定預算金額： AWS 預設單位為美金(USD)。若要設定台幣 100 元預算，建議在金額欄位輸入 3。
  3. 設定名稱與告警通知： 輸入名稱如 My-100-Budget（不能有空格），並綁定個人的電子郵件地址。
  4. 確認並創建： 當實際花費或預測花費達到預算金額的 80% 或 100% 時，AWS 就會自動發信通知警告。

☁️ 三、 從實體到虛擬：AWS EC2 雲端元件中英功能藍圖對照

本專案實作於 AWS 東京機房安全地建置一台 Windows Server 2025 虛擬機（EC2），其核心元件對照如下：

AWS 核心雲端元件 中英功能對照 實務技術解讀與運作原理
Instance EC2 虛擬伺服器實例 雲端上的虛擬主機，負責主要的程式碼運算與所有服務運行。
EBS (Elastic Block Store) 雲端區塊儲存體 / 虛擬硬碟 連接著 Instance，負責持久化存放系統檔案與核心資料。

Security Group 安全群組 / 虛擬防火牆 控制連接埠進出的第一道防線。本實作設定允許 RDP（遠端桌面連線）流量進出。

🌐 五、 網路工程基礎：URL 與 URN 的核心觀念對比

在學習雲端部署與協定架構時，必須釐清資源識別碼（URI）家族中，URL 與 URN 的差別：

比對特性 URL (統一資源定位符) URN (統一資源名稱)
核心概念 告訴系統資源「在哪裡 (Where)」，並指明具體的尋找路徑。 只負責「為資源命名 (What)」，確保唯一性，不包含存取路徑。
持久性與缺點 較低。如果檔案被移動或網域變更，URL 就會失效，進而產生 404 錯誤。 極高。具有永久性。就算檔案從 A 伺服器搬遷到 B 伺服器，它的 URN 依然保持不變。
